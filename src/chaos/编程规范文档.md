

### 玩家声明
格式: `@player`

### Git

单行 commit 格式: 
`<type>(<tag>,..) : <description>`

多行 commit 格式:
首行 `<type1> & <type2> & ..`
其他行 `<type1>(<tag>,..) : <description>`

#### type

| type      | 作用                                         |
| --------- | ------------------------------------------ |
| feat      | 新增 feature                                 |
| fix       | 修复 bug                                     |
| docs      | 仅仅修改了文档，比如 README, CHANGELOG, CONTRIBUTE等等 |
| style     | 仅仅修改了空格、格式缩进、逗号等等，不改变代码逻辑                  |
| refactor  | 代码重构，没有加新功能或者修复 bug                        |
| perf      | 优化相关，比如提升性能、体验                             |
| test      | 测试用例，包括单元测试、集成测试等                          |
| chore     | 改变构建流程、或者增加依赖库、工具等                         |
| revert    | 回滚到上一个版本                                   |
| chaos     | 过于混乱,交给description,也可省略description         |
| release   | 发布一个版本,\<description\> 为版本号                |
| nightly   | 每日更新                                       |
| merge     | 本次提交仅为合并分支                                 |
| clear     | 仅做清理工作(不影响代码), 例如: warn消除, 注释删除            |
| ignorable | 琐事工作,完全可忽略,用来代替过去使用的`...`                  |
| snapshot  | 提交 仅 用于 保存一个快照                             |
| format    | 仅格式化代码                                     |

#### tag
| tag     | 作用   |
| ------- | ---- |
| serious | 重大更新 |
| nightly | 每日更新 |

### Label
单行: `<lable>[<symbol>,..]: <description>`
多行:
```rust
/// #todo[if_pin] : <description>
///  , 空一格后 逗号开头,表示接着上一句描述
///  . 空一格后 句号开头开头,表示新描述
///  . if_pin : 若什么则删除label
```

`symbol`并非对所有`label`都可使用
部分`symbol`仅对部分`label`可用
且`label`可以单独 意义化 `symbol`的作用
`symbol`本身保持抽象,对于具体的`label`有具体的作用

| symbol     | 作用                                                          |
| ---------- | ----------------------------------------------------------- |
| consider   | 考虑中                                                         |
| wait       | 等待中                                                         |
| maybe      | 可能                                                          |
| pin        | 特意保留这个`label`,即使`label`的工作做完也不可删除这个`label`(除非对应的代码被移除)      |
| if_pin     | 特意保留这这个`label`,强调直到某些条件满足后删除(或对应的代码被移除时删除)                  |
| @player    | 声明这段标签的 编写者                                                 |
| burden     | 心智负担 声明,表示这会带来较重的心智负担, 往往与 \#node配合,例如 \#note\[burden]: ... |
| deprecated | 废弃的                                                         |


#### \#todo

##### \[consider\]
考虑中,不一定会去完成这个todo

##### \[wait\]
等待中,等待某些事件后,在完成的todo

##### \#todo/test
需要完成测试

##### \#todo/check 
需要完成一些检查工作
#### \#tag
`#tag(tag_name,..)`
用于 标记 , 与 对应的tag产生关联 , 追踪代码(debug用)

关联
在A代码中标记`#tag(abc)`
B代码中标记`#tag(abc)`
AB则产生关联

#### \#tip
一些提示,说明 ,例如说明 这块代码不必担心什么什么东西的发生
往往是 代码编写者 写个 其他代码阅读者 看的(包括 给未来的自己看)
有点类似 强调性注释
#### \#note
你需要特别注意的注释
#### \#plan
声明未来的计划, 往往意味着 大量代码的编写(模块,架构,..) (若只有小部分(函数,结构体字段,代码块..) 你应该用`#todo`)
#### \#perf
需要优化的标记

##### \[maybe]
可能可以这样优化,但不清楚 是否会产生 优化副作用(即本身优化成功,但影响了其他部分) 或 恶化(即错误优化,会导致性能更差)

#### \#err (未确定)
描述异常后会发生什么,等等

#### \#unsafe
声明 代码,描述 强调 不安全的地方

##### \#unsafe/ai
声明此处代码 被ai参与 或 完全由ai生成 , 无法确保安全性,故此标记

#### \#safety
声明 代码,描述 强调 只要按照描述做 即可保证安全性

#### \#bug
`#bug([scope]bug_number,...)` (语法规则暂未确定,但可使用目前的)

描述一段代码 会发生的bug (需具体描述,什么情况下会发生,什么环境下会发生,...)

fix作用域, 对于 bug和fix 影响的范围不同,需使用不同的,范围限定
例如: 对应的bug和fix,只在 fn范围 中出现 ,则使用 `fn` 范围限定,
若 bug 在 mod范围 出现 但 fix 在 fn范围 出现,则 取最大范围 `mod` 范围限定
若在 不同的 fn范围 或 mod范围, 需明确声明具体范围, 
例如 
bug出现在 mod1 , fix在 mod2::fn1
`#bug([crate::mod1,crate::mod2::fn1]9527)`
`#fix([crate::mod1,crate::mod2::fn1]9527)`
若父范围相同,可忽略具体范围声明

若无法确定,默认使用 `crate` 范围限定

若对于整个crate而言,可直接写 数字(一般四位数),例如 `#bug(0001)`

对于 单个`fn` 范围 ,可写 `#bug(fn_01)`

对于单个 `mod`范围,可写 `#bug(mod_01)`

##### \[maybe]
可能存在bug,需要注意

#### \#fix
`#fix([scope]bug_number,...)`
针对指定bug_number的修复描述

##### \[maybe]
可能使用这种方法进行修复bug (当无法确定这种方法是否可靠或正确时,可如此标记)

##### \#fix/reserve
以使用其他fix进行bug修复,但任保留这个fix方法 (但一定不使用, 注释也好,或者啥的,这个标记后,就一定不可在使用,这个方法去修复对应的bug)

#### \#deprecated
弃用的部分

##### \[maybe]
可能被弃用的部分

#### \#design
设计部分的声明

##### \[maybe]
可能会采用这个设计

##### \[consider]
对这个设计考虑中

##### \[deprecated]
废弃的设计

##### \[pin]
保留设计的说明

##### \[\<number>]
允许通过,number来关联,\#desing和\#design/effect

##### \[\<name>]
允许通过,name来关联,\#desing和\#design/effect


##### \#design/effect
被设计影响的部分


### Fn Documentation Comment
函数文档注释规范
```rust
/// <description: 函数的简介 必须在文档最顶端>
/// 
/// <parameters>
/// <returns>
/// 
/// <examples>
/// <test fn>
/// 
/// <safety>
/// <panic>
/// <note>
/// 
/// <label>
fn f(){

}
```

#### Label
标签需要紧贴函数上面
即总是在文档注释最底部
无描述Label,总是在最底部,且 并排

若 文档注释 不存在其他 标题,则可省略 Label 标题
```rust
/// # Label
/// <label1> : <description>
/// <label2> : <description>
/// <label3> , <label4>
fn f(){
}
```

#### Test Fn
声明 对该函数的测试用例在哪里
也可以直接在该标题下,写 测试
```rust
/// # Test Fn 
/// ```
/// # .. 测试代码 ..
/// ```
/// [`test_fn`](crate::tests::test_fn) : <description>
/// [`test_fn2`](crate::tests::test_fn2) : <description>
fn f(){
}
```

#### Examples
案例文档,教你如何使用这个函数
```rust
/// # Examples
/// ```rust
/// assert_eq!(1,1)
/// ```
fn f(){
}
```

#### Safety
特殊的安全性要求(若只有一条,可使用标签(`#safety`)代替)
```rust
/// # Safety
/// - 你不能做什么什么
/// - 你可以做什么什么
/// - 这是使用它的约定
/// - 这是使用它的前提
/// - 使用它后的注意事项
/// - ...
fn f(){
}
```

#### Panic
Panic触发条件的描述
```rust
/// # Panic
/// - 这样做会触发Panic
/// - 不这样做会触发Panic
fn f(){
}
```

#### Note
特别需要注意的提醒 (若只有一条,可使用标签(`#note`)代替)
```rust
/// # Note
/// - ...
fn f(){
}
```

#### Parameters
描述形参
```rust
/// # Parameters 
/// `a`: <description>
fn f(a:i32){

}
```

#### Returns
```rust
/// # Returns 
/// <description>
/// `.0` : <description>
/// `.1` : <description>
fn f()->(i32,u32){

}
```
